- [planted 2026-05-15] The "custom agent Dockerfile installs missing tool X" pattern is much more general than the p4-binary case from Track 2 lesson #2. The same recipe — `FROM jetbrains/teamcity-agent:latest` + RUN-install-tool — is how you'd add MSVC build tools, FASTBuild, Incredibuild agent, Blender for content cook, Houdini, etc. to an agent image. Track 2 step 3 (sample C++ project) and Track 3 (build acceleration / FASTBuild) both want this pattern. Worth a small `ci/agent/` subdir holding one Dockerfile that grows over time, with each tool-add accompanied by a one-line note in the agent README explaining *why* that tool is there. That doc-as-you-go discipline is the only thing that keeps an agent image from rotting into "we installed X in 2023 and nobody remembers why."
- [planted 2026-06-03] The `accel/scripts/demo-mp.ps1` pattern — generate N heavy TUs, compile two ways, best-of-3 cold reps, report a speedup table — is a reusable *build-acceleration measurement harness*, not a one-off. Every remaining Track 3 lever is the same harness with one variable changed: unity/jumbo = concatenate the N TUs into one before compiling; PCH = `/Yc` once + `/Yu` per TU vs no-PCH; FASTBuild = the fbuild orchestrator vs raw cl. Worth factoring the timing/reporting core out of demo-mp into a shared helper so each new lever is ~20 lines + a README row. The interview value compounds: one consistent before/after table across /MP, unity, PCH, FASTBuild tells a complete "how I'd cut a 25-min build to 12" story with real numbers, not anecdotes.
- [planted 2026-06-03] The agent-pool benchmark (`bench-agents.ps1`) shows a clean 2x, but on this toy project every build is ~11s of *fixed overhead* (p4 sync + artifact download + handshake) - the 2x is overhead-overlap, not compute-parallelism.
